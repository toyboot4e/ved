# Plan: user extensions

Status: **implemented** (2026-07) — all nine steps landed; the work list
below records what each shipped and the deliberate deferrals (anchored
overlays, third-party extension options, a published types package, panel
sides beyond bottom). `test/e2e/user-extensions.ts` exercises the whole
surface end to end. Next: fold the durable parts of this design into
`docs/extensions.md` and `docs/architecture.md`, then shrink this file to
the deferrals.

Goal: let a user extend and configure ved — commands, keybindings, editor
behavior, shell UI — without cloning the repo or setting up a package. One
concept, **extension**, in three packaging tiers that share a single
contract. The word "plugin" is not used anywhere (one concept, one word).

## Terminology and layering

- **Extension** (user-facing): a unit of user code ved loads. Packaged as
  (a) an in-repo workspace package (today's tier — `@ved/vim`), (b) a single
  `.ts` file, or (c) a directory with a manifest. Same contract in all three.
- **The editor seam** (`EditorExtension` / `EditorExtensionContext` in
  `@ved/editor`, per `docs/extensions.md`): the internal mechanism. User
  extensions never construct one; the loader wraps their editor hooks into
  one per open editor.
- **`VedContext`** (app-level, new): what a user extension's `activate`
  receives. Wraps the editor seam for the focused editor and adds the shell
  surfaces (keybindings, UI, storage) the seam deliberately lacks.

## Configuration is code

There is no data config file. User configuration IS an extension:
`init.ts` in the config dir is an ordinary single-file extension with a
conventional name — it binds keys, sets options, registers small commands.
One surface to learn, one way to bind a key, and the typing story (generated
`ved.d.ts`, below) makes it self-documenting.

- **Keybindings**: `ctx.keybindings.bind("Mod+K", "my-ext.reflow")` feeds
  the single `Chord → command id` table `commands.ts` already isolates for
  exactly this swap (ids → semantics → bindings, three separate layers).
  Any extension may bind (shipping defaults); load order is deterministic
  (sorted, `init.ts` LAST) and later bindings win with a notice — so the
  user's own bindings always take precedence. The chord vocabulary is
  `Mod+…` only (`chordOf`); plain-key and multi-stroke bindings are
  `handleKey` extension territory (Vim's lane).
- **Machine-owned app state** (window geometry, session restore, persisted
  menu toggles) is NOT user configuration: it lives in a separate
  ved-written file under `userData` that users never edit, so "the app
  remembers a toggle" never requires machine-writing user code.
- Deferred, not designed here: configuring a *third-party* extension from
  `init.ts` (the Emacs variables-before-load story). V1 user extensions are
  mostly the user's own code; revisit when a real third-party extension
  needs options. If ved ever wants a GUI settings editor, a thin data layer
  can slot UNDER code config then (data defaults, code wins) without
  unwinding this design.

## The config directory

Default: the platform config dir (`~/.config/ved` on Linux). Overridable
with the CLI flag `--config-dir <path>`, which the main process resolves
once at startup and everything else (loader scan, generated types, storage)
derives from. The flag is also the test seam: e2e drivers point it at a
temp dir to exercise the extension system with fixture extensions and no
doc-clearing, the same isolation move as `VED_SMOKE_USER_DATA`.

Layout:

```
<configDir>/
  extensions/
    init.ts            # the user's config — a single-file extension, loaded last
    reflow.ts          # single-file extension (id: "reflow")
    word-count/        # project extension (id from its manifest)
      package.json     #   { "ved": { "id": "…", "entry": "src/main.ts", "minAppVersion": "…" } }
      src/…
    ved.d.ts           # generated by ved on startup — do not edit
    tsconfig.json      # generated by ved on startup — do not edit
```

## The loading pipeline

Extensions run in the **renderer** (they drive the editor), but the renderer
has no fs and no TypeScript. So:

1. **Main** scans `<configDir>/extensions/`. A `*.ts` file is a single-file
   extension; a subdirectory with a `package.json` is a project extension
   (entry from its `ved` field).
2. Main compiles to one JS string per extension: `esbuild.transform` for a
   single file; `esbuild.build` (`bundle: true, external: ["ved"], write:
   false`) for a directory — which buys multi-file sources and npm
   dependencies with no user-side build step. Cache by content hash.
3. The JS string crosses the typed IPC to the renderer, which imports it as
   a module: `Blob` → `URL.createObjectURL` → `import(blobUrl)`. An import
   map in the renderer HTML resolves the bare specifier `"ved"` to the
   module exposing `VedContext`'s types and nothing else (CSP gains
   `script-src blob:`).
4. The module exports `activate(ctx: VedContext)` (and optionally
   `deactivate()`). Loader wraps every call in try/catch; a broken extension
   surfaces a notice and is skipped — it never takes down the editor.

Hard boundary this pipeline creates (a feature, not a limitation): extension
code runs sandboxed in the renderer — no Node, no Electron globals. Bundled
dependencies must be browser-safe (`fs`/`net` imports fail at bundle time).
An extension that legitimately needs fs gets a capability on `VedContext`
(`ctx.storage`, backed by the existing `ipc.ts` contract), never Node.

**Typing without setup**: on startup ved writes `ved.d.ts` (generated from
the real API — cannot drift) and a minimal `tsconfig.json`
(`paths: { "ved": ["./ved.d.ts"] }`) into `extensions/`. Any editor then
resolves `import … from "ved"` with full types — no npm install, no
package.json for the single-file tier. Project extensions may instead depend
on a published types-only package once one exists. Type *stripping* only:
TS syntax compiles, but nothing type-checks at load time and `enum`/emitting
decorators are out — fine for user scripts; the user's own editor is the
type checker.

**Dev loop**: a `ctx`-independent `dev` list (CLI flag `--dev-extension
<path>`, repeatable) points at working directories. Main watches the entry
graph (esbuild reports it), re-bundles on change, and reloads: dispose
everything the old instance registered, import the new blob. Disposal is
automatic — see namespacing. `init.ts` itself hot-reloads the same way.

## Namespacing — by construction, not convention

Today `registerCommand` accepts any id and silently replaces existing ones;
"namespace your ids" is a comment. The loader turns it into a guarantee:

- The bare `"ved"` import exports **types and pure helpers only**. Every
  capability lives on the `ctx` handed to `activate`, and that `ctx` is
  bound to the extension's id (manifest `ved.id`, or the filename for
  single-file extensions). `ctx.commands.register("reflow", …)` registers
  `<id>.reflow` — the prefix is applied by ved, not typed by the author.
  There is no unprefixed registration API to misuse. Core namespaces
  (`appear.*`, `history.*`, `caret.*`, …) are thereby unreachable; duplicate
  extension ids are a load error.
- *Executing* foreign commands is allowed (`ctx.commands.execute("history.undo")`)
  — composition is the point of commands; `ctx.keybindings.bind` likewise
  takes any full command id. Registering into a foreign namespace is not.
- The same prefix applies to CSS (`setContentClass` classes, decoration
  classes) and DOM contributions (`data-ved-ext="<id>"`).
- Every registration returns a `Disposable` and is recorded on the context,
  so deactivate / crash / hot reload sweeps exactly that extension's
  commands, bindings, hooks, listeners, and UI. No extension-authored
  cleanup bookkeeping.

## `VedContext` sketch

The editor seam is already `ctx.editor` nearly verbatim — plain text, plain
offsets, movement, caret stops, IME safety enforced by the seam. The sketch
below annotates what is new versus delegated.

```ts
// `import "ved"` exposes types only; capabilities arrive via activate(ctx).
export function activate(ctx: VedContext): void | Promise<void> {}
export function deactivate(): void {} // optional; disposables auto-clean

type VedContext = {
  readonly extension: { readonly id: string; readonly version: string };

  // Wrapper over the seam's registerCommand/runCommand, prefix enforced.
  readonly commands: {
    register(name: string, run: (args?: unknown) => boolean | Promise<boolean>): Disposable;
    execute(id: string, args?: unknown): Promise<boolean>; // any full id
  };

  // Feeds the single Chord → id binding table (commands.ts). Any full id;
  // later registrations win with a notice (init.ts loads last).
  readonly keybindings: {
    bind(chord: Chord, commandId: string): Disposable;
  };

  // Delegates to the FOCUSED editor's EditorExtensionContext, 1:1 names:
  // text/selection/setSelection/replaceRange/moveCaret/moveCaretVisual/
  // caretStop/snapCaret/deleteStep/scrollPage/setCaretShape/setContentClass/
  // setVisualSelection/breakUndoGroup/isComposing — same offset semantics.
  readonly editor: SeamDelegates & {
    /** Key/text/composition hooks, wrapped by the loader into an
     *  EditorExtension attached to every open editor (stable identity
     *  managed by the loader). Contract of docs/extensions.md applies:
     *  return false for chords you don't own. */
    addHooks(hooks: EditorExtensionHooks): Disposable;
    /** NEW editor-core surface: plain-offset ranges in, view-only
     *  decorations out, applied on the editor's IME-safe schedule.
     *  Declarative — each call replaces the extension's set. */
    decorate(ranges: ReadonlyArray<{ from: number; to: number; class: string }>): Disposable;
    onDidChangeText(fn: (text: string) => void): Disposable;      // taps onTextChange
    onDidChangeSelection(fn: (sel: SelectionOffsets) => void): Disposable;
  };

  // Shell surfaces (renderer, outside the editor core).
  readonly ui: {
    statusItem(init: { text: string; title?: string; onClick?: () => void }): StatusItem;
    panel(init: { title: string; side: 'left' | 'right' | 'bottom' }): Panel; // .element owned by the extension
    notice(msg: string, kind?: 'info' | 'warn' | 'error'): void;  // → notice.ts
    quickPick<T>(items: T[], opts: { label(item: T): string }): Promise<T | null>; // → quick-open machinery
  };

  readonly storage: {   // per-extension dir under <configDir>, via ipc.ts
    read(file: string): Promise<string | null>;
    write(file: string, data: string): Promise<void>;
  };
};
```

## Editor UI: two doors

- **Shell surfaces** (`ctx.ui`) are free rein: for panels the extension owns
  a container element outright (framework-agnostic by construction); status
  items and notices are plain data.
- **Inside the editor** is declarative only. Extensions never see the
  ProseMirror view or the contentEditable DOM. `decorate()` speaks plain
  offsets and lands as view-only decorations — the same category as
  bold/italic (architecture.md: every non-ruby format is a decoration), so
  an extension *cannot* make displayed text diverge from model text: the API
  has no vocabulary for it. Updates apply on the editor's IME-safe schedule;
  a call mid-composition queues. Anchored overlays (pin an element near
  offset N) are deliberately OUT of scope for v1 — exposing rect queries
  invites unscoped glyph walks (the per-caret-move perf invariant) and
  multicol geometry is subtle enough that extensions would hold them wrong.

## Multi-buffer rule (v1)

The seam is per-editor-instance; the desktop is a multi-buffer tabbed shell.
V1 rule, matching how `vim.ts` wires in today: the loader attaches every
extension's hooks to every open editor; `ctx.editor` methods target the
focused one. A `ctx.workspace` (enumerate buffers, open files via IPC) is a
later rung, not v1.

## Work list

Each step lands with `just test-all` green and an e2e driver exercising the
new behavior (extensions with UI need one — `vim-mode.ts` is the template;
any new dialog needs its `VED_SMOKE_*` seam).

1. ✅ **Config dir resolution** — `--config-dir=<path>` (equals form only:
   a space-separated value would read as a positional file argument) in
   `main/config-dir.ts`, platform defaults, resolved once in `main/index.ts`;
   doubles as the e2e isolation seam (`test/e2e/user-extensions.ts`).
2. ✅ **Loader MVP** — single-file tier: `main/extension-host.ts` compiles
   with Node's own `stripTypeScriptTypes` (no esbuild dependency until the
   project tier needs bundling), `init.ts`-last order, reserved-id refusal,
   generated `ved.d.ts` (the raw source of `shared/extension-api.ts` — the
   types-only single source both the declaration and the renderer
   implementation come from) + `tsconfig.json`; blob import + id-bound
   `VedContext` in `renderer/src/extension-host.ts`; per-extension
   registration tracking so a failed `activate` sweeps clean. The `ved`
   module is types-only, so no import map is needed at all — `import type`
   strips away with the types (verbatimModuleSyntax makes the user's checker
   enforce it); CSP gains `script-src blob:`.
3. ✅ **`ctx.keybindings`** — per-chord binding stacks feed the editor's
   `keybindings` prop merged over `DEFAULT_KEYBINDINGS`; shadowing another
   extension's binding notices (rebinding a default is silent — that is the
   normal use); disposal restores the previous binding.
4. ✅ **Editor wiring** — `addHooks` fans out through one loader-built
   `EditorExtension` per user extension (guarded: a throwing hook reports,
   never breaks the key path); `onDidChangeText` taps the shell's
   `onTextChange`; `onDidChangeSelection` rides the payload-free
   `onSelectionChange` PING prop (editor.tsx) — listeners pull offsets
   lazily through the seam, so caret moves stay O(1) with none.
5. ✅ **UI surfaces** — `notice`, `statusItem` (footer right edge, absolute
   so the centered counter is untouched), `quickPick` (the quick-open
   overlay's little sibling: same styles, own store, overlay key scope in
   keymap.ts, one at a time — a new picker preempts with null), `panel`
   (bottom-docked above the shell panel; the extension OWNS the body
   element, alive across show/hide). Panel sides beyond bottom: deferred.
6. ✅ **`decorate()`** — the one editor-core addition: a `setDecorations`
   seam method whose ranges fold into the decoration BASE cache exactly like
   the search highlights (identity-keyed — an idle set costs caret moves
   nothing; `caret-move-perf`/`click-perf` still count 0 rebuilds).
   IME-safe by construction: mid-composition only the ref updates; the
   composition's own commit transaction repaints. Classes are namespaced by
   the host (`vedx-<id>-…`), background-only by contract. Anchored overlays
   remain OUT (unscoped rect walks).
7. ✅ **Project tier** — `package.json` `ved: { id?, entry, minAppVersion? }`,
   `esbuild.build` bundling (relative imports + browser-safe npm deps; `ved`
   deliberately NOT external — a value import fails loudly at bundle time),
   `--dev-extension=<path>` links, and watch → recompile → renderer hot swap
   (deactivate + automatic sweep, same load-order slot) for ALL extensions,
   init.ts included. No content-hash disk cache — bundling is ms; revisit if
   startup ever shows it.
8. ✅ **Storage** — `ctx.storage` read/write under `<configDir>/storage/<id>/`
   behind ipc.ts, id-bound in the host (no cross-extension access by
   construction), single-segment names only (traversal-guarded in main).
   The machine-owned app state (geometry, session restore) already lives
   apart from user code (renderer localStorage); moving it under the config
   dir was judged not worth the churn — revisit if it should survive
   profile switches.
9. ✅ **Hardening** — the seam's `registerCommand` refuses to shadow a CORE
   command id; reserved/duplicate extension ids are load errors;
   `minAppVersion` gates project extensions.

## Known gaps this plan closes

- `registerCommand` trusts callers to namespace and silently replaces
  existing ids (extension-context.ts) — steps 2 and 9.
- The seam has no document/selection *events*; extensions only hear keys and
  compositions — step 4.
- No decoration surface; styling levers are caret shape, content classes,
  visual-selection kind only — step 6.
- In-repo packaging friction (workspace entry, tsconfig references, vitest
  projects, Biome override, Nix hash) is the cost this plan lets user
  extensions skip; it remains the right tier for first-party extensions
  like `@ved/vim`.
